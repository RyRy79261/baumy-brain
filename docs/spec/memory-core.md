# Memory Core — schema-light, memory-first (the product heart)

> Baumy "keeps track like a well-trained human" without anyone defining categories or knowing a database exists. This section specifies the invisible tiered-memory substrate that makes that true: how arbitrary things get remembered, retrieved, kept fresh, attributed to who said them, and defended against poisoning — all on the locked stack (Neon Postgres + Drizzle + pgvector, Vercel Hobby $0, Vercel AI SDK, Inngest for async). It is **ONE shared house pool** (Baumy is a house-management tool, not a personal PA), every item **author-attributed**, keeping **both the verbatim transcript and a derived knowledge graph**, with a **"secure value"** sensitivity flag protected by **app-side encryption**.

## Overview

The memory core is a **domain-neutral substrate** that encodes the *shape* of memory (who said it / what / about-whom / when / still-true?) and **never the subject**. "Domain" — groceries, guests, a friend's event run out of the house, 3D-printing, a sink rebuild — is never a table, enum, or column. It is emergent from (a) a free-text soft label (`TEXT`, never `pgEnum`), (b) a natural-language `content` string, and (c) a vector embedding that makes anything semantically retrievable even when it was never structured.

**One shared house pool, author-attributed (A3 / A3b / B10).** There is a single house memory — **no per-user private memory, no `visibility` column, no `owner_user_id`, no RLS**. Instead every memory row and every fact carries **who authored it** (`author_tg_id → member → name`), so Baumy can answer *"what did Tom say about the landlord?"* and cite who/when. Privacy is handled as a **"privacy discretion" behavior** (soft-redaction judgment reusing the sensitivity scanner on *public* replies), **not** as hard private/shared silos. Group membership *is* the roster: a first message from a new `user_id` auto-creates a member; a member who leaves is deactivated but their contributed memory stays.

**Three cooperating layers** (retention decision D17 — keep BOTH verbatim and graph, embed both):
- **Verbatim transcript layer — `source_messages`**: every message stored **in full text, retained forever** (full text + author + timestamp). This is the evidence/quote layer *and* a **bot-queryable transcript** that fixes Telegram's no-scrollback limitation — Baumy searches its own copy (it still cannot read Telegram history it never received). Trivial storage at house scale; **never nulled** (supersedes the old "null raw after processing" lean).
- **Evidence embedding layer — `memory_items` (+ `memory_embeddings`)**: every salient message becomes an embedded natural-language row. The semantic safety net — retrievable by meaning even when the extractor fails to structure it.
- **Knowledge layer — `entities` + `facts`/`edges` (+ `fact_embeddings`)**: distilled, canonical, current-state assertions with bitemporal validity + provenance + author, **also embedded** so semantic search hits derived facts too. Grounds replies, answers "what's coming up / what did X say," and points back to `source_messages` for exact quotes.

**Dated events + recurrence live in the substrate (A4).** Facts carry an optional `event_at` and free-text `recurrence` (guest arrival, a friend's event date, an annual/monthly bill). A scheduled Inngest scan surfaces upcoming dated facts (advance notice ~a week) to feed digests, conservative nudges, and event-anchored reminders. This is distinct from the reminder *rows* themselves.

**Origin-scope for the future (D-sec).** Every memory-bearing row carries a **`group_id` origin-scope** column. v1 is a single house group (`BAUMY_HOUSE_CHAT_ID`), so it is constant; retrieval always filters by it so multi-group / a second house is an **additive flip later, not a rewrite**. Multi-tenant SaaS remains a NON-GOAL.

**Secure values (D-sec).** A `is_secure` sensitivity flag marks genuinely-secret items (door/gate/alarm codes, wifi, bank details). Their literal value is **encrypted app-side** (the bot holds the key in an env/Vercel secret) before write and decrypted only to answer a house member's direct request — a DB dump alone is useless. Secrets are **never volunteered unprompted, never embedded in plaintext, never included in digests/broadcasts.**

**Reminders are the one hard-structured, scheduled exception** (see `data.md` / `llm-pipeline.md` / Inngest); they are **house-scoped and delivered to the group**, not per-user DMs. Everything else lives in the soft substrate.

**Build, don't buy.** No memory framework (mem0, Letta/MemGPT, Zep+Graphiti, LangMem, Cognee, Supermemory) fits the locked constraints without dragging in forbidden infra (a dedicated graph DB, an always-on server, or a third-party cloud). We **build a thin in-process module** on Neon+Drizzle+pgvector and **borrow mem0's published two-phase algorithm as a design pattern** (extraction → reconcile), implemented ourselves. Clean-room only: reuse mem0's Apache-2.0 prompt text as reference, never its runtime. This dual layer (pgvector + relational graph, both in Postgres) is the **ceiling of fancy** at 4-person scale — do NOT overbuild toward Neo4j / heavy GraphRAG.

## Concrete design

### Tables (domain-neutral primitives + evidence layer)
- **`source_messages`** — provenance/dedupe anchor **and verbatim transcript**. `update_id` UNIQUE (Telegram monotonic per-bot int), `(chat_id, message_id)`, `author_tg_id`, `group_id` (origin-scope; = `chat_id`/house in v1), `is_trusted` (false for all group text), **`text TEXT NOT NULL` (full verbatim message, retained forever — never nulled)**, `created_at`. Every memory row FKs here. Edited messages re-run the full gate and supersede (D18) — store the edit as a new/updated verbatim row, never trust an edit more than the original.
- **`entities`** — deduplicated nodes. Free-form `kind TEXT`, `canonical_name`, `aliases[]`, `name_embedding vector(1536)` for fuzzy resolution, `group_id` (origin-scope).
- **`memory_items`** — the embedding substrate (evidence net). Free-form `memory_type TEXT`, `content TEXT` (source-of-truth), `author_tg_id` (denormalised from `source_messages` for author-boosted retrieval), `group_id`, `salience`, `access_count`, `last_accessed_at`, `is_active`, `source_ref` (FK `source_messages`). Embeddings held in a **separate `memory_embeddings` table** (1:N, `UNIQUE(memory_item_id, model)`, `embedding vector(1536) NOT NULL`) so items can be written synchronously on ingest (neon-http) and embedded later inside an Inngest step, and re-embedded on model change.
- **`facts`** — distilled current-state. Unify attribute-facts and relationships in one table: `object_entity_id` set ⇒ entity-to-entity **edge**; `object_value`/`object_json` set ⇒ literal attribute. Free-form `predicate TEXT`; bitemporal columns (`valid_from`/`valid_to`/`recorded_at`/`invalidated_at`); `is_current`, `supersedes_id`, `trust_tier`, `source_ref`; **`author_tg_id`** (who asserted it); **`group_id`** (origin-scope); **`event_at TIMESTAMPTZ NULL`** + **`recurrence TEXT NULL`** (dated-event + recurrence, e.g. `'annual'`/`'monthly'`/RRULE); **`is_secure BOOLEAN NOT NULL DEFAULT false`** + **`object_value_encrypted TEXT NULL`** (app-side ciphertext for secure facts; plaintext `object_value` NULL when `is_secure`).
- **`fact_embeddings`** — mirrors `memory_embeddings` for derived facts (1:N, `UNIQUE(fact_id, model)`, `embedding vector(1536) NOT NULL`). For secure facts, embed the **non-secret descriptor only** (e.g. "the front-door code") — never the secret value.

### Author attribution & discretion
- `author_tg_id` resolves to a member (`members.telegram_user_id → name`, table owned by `data.md`/auth). Members are **auto-discovered** from group activity (B10); a member who leaves is deactivated but their memory rows remain.
- Retrieval accepts an optional **author filter/boost**: when a query names a person ("what did Tom say…"), resolve name → `telegram_user_id` and boost/pre-filter matching rows.
- **Privacy discretion** (not a partition): before a *public* reply, the outbound sanitizer applies soft-redaction judgment (reuse the sensitivity scanner) — e.g. don't broadcast a door code — but the underlying pool is shared and there is no private lane.

### App-side encryption (secure values)
- Key: `BAUMY_ENCRYPTION_KEY` (32 bytes, base64 in a Vercel/env secret — NOT in the DB). Verify/pin at build.
- Cipher: AES-256-GCM via Node `crypto` — `createCipheriv('aes-256-gcm', key, iv)`; store base64(`iv(12) || authTag(16) || ciphertext`) in `object_value_encrypted`.
- Write path: when the extractor flags `is_secure`, encrypt the value before insert; leave `object_value` NULL; embed only the descriptor.
- Read path: decrypt only to answer a direct member request; **never** in digests/broadcasts/logs. Losing the key = unrecoverable secrets (accepted trade-off; key rotation deferred).

### Vector + retrieval
- Enable pgvector as the **first, hand-written migration** — `drizzle-kit generate` emits `vector(1536)` columns but **never** `CREATE EXTENSION`, so a fresh DB fails without it. Also enable `pg_trgm` (typo/nickname entity matching). **`pgcrypto` is NOT required** for secure values — encryption is app-side (D-sec); only enable it if another consumer needs it (see `data.md` migration 0000).
- Drizzle: `vector('embedding', { dimensions: 1536 })`; index via the array-return table callback `(t) => [ index('...').using('hnsw', t.embedding.op('vector_cosine_ops')) ]`; query with `sql\`1 - (${cosineDistance(col, queryVec)})\``.
- **Retrieval score** = `w_sim·cosine + w_rec·recency + w_imp·importance + w_trust·trust_norm (+ w_author·author_match)`, with hard pre-filters. **Always pre-filter by `group_id`** (constant in v1, defense for multi-group later). Two modes: (a) conversational grounding — `is_active`, untrusted rows allowed as *context only*; (b) privileged mode — trusted-only, feeds the write-gate/notification path. A third **transcript mode** searches `source_messages.text` (the bot-queryable transcript) for exact quotes. Over-fetch then re-rank; always UNION structured current-fact lookups (over `facts` + `fact_embeddings`).
- **Temporal decay** reinforced by *last access*, not creation: `strength = importance · exp(−ln2 · hours_since_last_reinforced / half_life)`; half-life scales with importance so useful memories persist and one-off trivia ages out.
- **Upcoming-events retrieval**: a helper over `facts WHERE event_at IS NOT NULL` (expanding `recurrence`) returns dated facts within a lead window (~7 days) for digests/nudges/event-anchored reminders.

### Ingestion pipeline (all heavy work on Inngest)
1. Webhook dedupes on `update_id`, inserts `source_messages` **with full verbatim `text`** (`is_trusted=false` for group; `group_id` from the house chat), returns 200 fast, emits `baumy/message.ingested`.
2. **Salience gate** (cheap OpenAI nano): `skip | extract`. Load-bearing — with privacy OFF, Baumy sees *every* message; without this gate the store fills with "lol"/"ok." (Verbatim `source_messages` is still kept regardless — the gate only governs extraction/embedding.)
3. **Extraction** (structured output): atomic candidate `{entities[], facts[], edges[]}`, each carrying `author_tg_id`, trust tag, and — where present — `event_at`/`recurrence` and an `is_secure` flag.
4. **Reconcile** in a *pooled* transaction (neon-serverless — neon-http has no transactions): embed → cosine top-k within same `group_id` AND `invalidated_at IS NULL` → LLM chooses ADD/UPDATE/DELETE/NOOP → soft-supersede on update; **app-side encrypt** any `is_secure` value (plaintext dropped, descriptor embedded); write `memory_embeddings` + `fact_embeddings`.
5. **Offline sweeps** (scheduled Inngest): decay/expire, dedupe/merge near-duplicate entities, contradiction resolution, quarantine sweep, **dated-event scan** (surface `event_at` within the lead window → emit event for digests/reminders).

## Gotchas

- **`CREATE EXTENSION` is not emitted by drizzle-kit** — hand-write it as migration 0000 (`vector`, `pg_trgm`) or the first `vector(...)` migration fails on a fresh Neon DB. `pgcrypto` is no longer needed here (encryption moved app-side).
- **`drizzle-kit push` bug #5792** regenerates the HNSW index **without** the operator class (drops `vector_cosine_ops`) → migration fails / wrong index. Use `generate` + `migrate` only; add a CI drift gate; never `push` to prod.
- **HNSW 2000-dim cap**: you can never swap in `text-embedding-3-large` (3072) on the plain `vector` type without moving to `halfvec` or dimension reduction. Lock the model+dim in one shared constant and write an idempotent re-embed backfill up front (applies to `memory_embeddings` AND `fact_embeddings`).
- **`hnsw.ef_search` defaults to 40** (low recall — Baumy "forgets"). `SET hnsw.ef_search = 100` per session (pooled driver), over-fetch, re-rank.
- **neon-http has no transactions** — supersession (flip `is_current` + insert) and multi-row writes MUST use the pooled (neon-serverless / WebSocket) driver.
- **Telegram ids are 64-bit and negative for groups** (supergroups `-100…`). Store `chat_id`/`message_id`/`update_id`/`author_tg_id`/`group_id` as `bigint`/`text`, never `serial`/`int`.
- **Cosine correctness needs normalised vectors** — OpenAI embeddings are L2-normalised, so `vector_cosine_ops` is right; never mix in unnormalised vectors.
- **The "zero schema change" promise only holds if `entity_type`/`predicate` stay TEXT.** The moment someone "tidies up" into pgEnums, every new domain reintroduces a migration. Guard with column comments + an ADR + a CI grep.
- **Verbatim retention is a real surveillance surface, but owner-accepted (D17).** Full text is kept forever — do NOT null it. Mitigate by scope: access is house-members-only (single-tenant, no third-party hosting), not by deletion.
- **Secure values must never leak sideways.** Never embed the plaintext (embed only a descriptor), never log it, never place it in a digest/broadcast, and decrypt only to answer a direct member request. A missing/rotated `BAUMY_ENCRYPTION_KEY` makes existing secrets undecryptable — treat the key as a durable secret.
- **Denormalise `author_tg_id` onto `memory_items`/`facts`.** Author-boosted retrieval ("what did Tom say") should not require a join back to `source_messages` on the hot path; keep author (and `group_id`) on the row.
- **Always filter retrieval by `group_id`.** It is constant in v1, so forgetting it "works" — until a second group exists and cross-house leakage appears. Bake the filter in now.
- **Do NOT attribute forwarded/bot content to a house member.** `author_tg_id` comes from the Telegram `from` field of a *native* group message; forwarded or bot-origin text stays untrusted and is not credited to a housemate (injection wall — see `security.md`).

## Risks & mitigations

| Severity | Risk | Mitigation |
|----------|------|-----------|
| High | **Prompt-injection / memory poisoning** — group text fabricates facts or tries to steer privileged writes/config. | Deterministic trust tier at ingest; untrusted rows are retrieval-only and can never supersede/merge/delete trusted facts or drive notifications/config; write-gate in `packages/core`. Group membership = baseline trust only, never privileged. See `security.md`. |
| High | **Secret leakage** — a `is_secure` value exposed via embeddings, digests, logs, or a broadcast. | Encrypt app-side (key in env, not DB); embed only a non-secret descriptor; disclosure discretion (answer only a direct member request, never volunteer, never in digests/broadcasts); scrub from logs. |
| High | A maintainer "improves" the schema into pgEnums / per-domain tables / a `visibility` column, killing the no-categories + shared-pool properties. | Column comments + ADR + CI grep guard failing on pgEnum over `memory_*` labels, new domain-noun tables, or a `visibility`/`owner_user_id` column. |
| High | High-volume group chat floods the store with low-salience noise → retrieval precision drops. | Mandatory cheap-model salience gate before embedding; bitemporal invalidation drops superseded facts; scheduled consolidation/decay. (Verbatim `source_messages` is exempt — it is the cheap evidence layer, kept regardless.) |
| Medium | **Author misattribution** — forwarded/bot content credited to a housemate, or nickname→member resolution wrong. | Attribute only native group `from` ids; forwarded/bot content stays untrusted and uncredited; entity/member resolution uses trgm + name-embedding with a conservative threshold. |
| Medium | Entity-resolution drift fragments one thing ("the printer"/"Prusa"/"MK4") across rows. | Match-then-merge in reconcile using name-embedding cosine + alias/trgm overlap with a conservative threshold; prefer new entity over false-merge; periodic merge sweep. |
| Medium | Embedding model/dim lock-in (2000-dim HNSW cap + hard-coded 1536), now across two embedding tables. | Centralise model+dim in one constant; separate `memory_embeddings`/`fact_embeddings` keyed by model; idempotent Inngest backfill; consider `halfvec` if larger models ever needed. |
| Medium | Per-message extraction+embedding token cost far exceeds "4-user" expectations. | Heuristic pre-filter → nano "worth remembering?" gate → batch per turn; `embedMany` batching; embedding both messages AND facts is bounded by the salience gate. |
| Low | Verbatim retention is a standing surveillance surface. | Owner-accepted (D17); mitigate by scope — single-tenant, house-members-only access, `group_id` origin-scope; no third-party hosting. |
