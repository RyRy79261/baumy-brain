# Baumy Brain — Risk Register

> Consolidated and deduplicated across the section specs, ranked most-severe first, and **reconciled to `00-decisions.md`** (the owner's finalized decisions supersede the earlier spec wherever they differ). "Owner?" flags a risk whose handling still needs a decision from you (cross-ref `00-open-questions.md` and the per-section open questions); the rest are engineering mitigations to build in.
>
> The defining fact that generates most of the top risks is unchanged: **Telegram privacy mode is OFF, so every group message is untrusted, attacker-controlled input** that flows into the LLM and can reach memory, notifications, config, and (without gates) privileged actions.
>
> v1 scope now adds four capabilities that each widen that surface — a **deliberative web-search lane** (the only path that reaches an external network), **user-defined scheduled tasks** (standing, paid, recurring jobs), **natural-language self-config** of the response/nudge policy (which an injection could use to *mute* Baumy), and **app-side encryption** of a flagged secret subset (a new key to lose or leak). Each bolts onto the existing spine rather than bypasses it (see the through-line).
>
> **Dropped from the register with their scope (per `00-decisions.md`):** the private-DM notification lane (A3 — house content is group-only, so the old "private content leaks to the group when a housemate never `/start`ed" risk is largely gone); per-user private memory / **row-level security** (A3/D9 — RLS is inert in v1, house memory is one shared pool); the curated `BAUMY_ALLOWED_TELEGRAM_USER_IDS` env allow-list (B10 — group membership *is* the roster); and Google-OAuth / email magic-link login and its redirect-callback plumbing (A1b — login is a Telegram bot-DM magic link only).

| 6 | **Hallucinated house-state** — Baumy invents a chore/guest/event/"who said what" it never stored, destroying the trusted-secretary persona (worst in the sparse first week; cold-start memory starts empty since bots can't backfill group history). | Critical · High | **Ground every recall strictly in retrieved provenance-bearing rows**, never raw chat recall; similarity floor + explicit honest-miss path ("I don't have anything on that"); render empty retrieval as an explicit empty block; cite `observed_at`/confidence/author; regression tests on the empty/low-confidence case. | — |

## Tier 2 — Availability & cost (the ~$0.50/day budget is a hard constraint)

| # | Risk | Sev · Likelihood | Mitigation | Owner? |
|---|------|------------------|-----------|:---:|
| 9 | **Denial-of-wallet / cost blow-up** — privacy-off means every message could trigger a paid classify/reply; unbounded RAG retrieval (input is a direct cost multiplier); or the expensive tier reached by a misclassified message (~$10–30/mo). | High · Med | Layered: deterministic pre-filter (~50% drop) → cheap nano classify → Claude only on grounded replies/digests. Durable Postgres spend ledger with a `~$0.50/day` cap → degraded mode; per-request token logger; reminder delivery never gated. | ✅ (model tier / cap) |
| 10 | **Scheduled-task runaway cost / infinite loops** (new capability) — a member stands up many high-cadence web-search/Opus tasks, or an injected "we're not done, keep searching daily forever" planted in memory is read by the completion evaluator and keeps a paid recurring job alive indefinitely. | High · Med | Creation is **write-gated** (no group-text creation; owner full / trusted confirmed); `BAUMY_SCHEDULED_TASK_MAX_ACTIVE` global cap enforced at create; per-run `MAX_WEB_SEARCHES_PER_RUN` + output-token budget; **every task carries a hard `until_at`/`max_runs` backstop** — the model-judged completion predicate can only *stop early*, never extend past the hard bound (completion input framed as untrusted DATA + audited); the `llm_budget_day` cap **defers, never drops** the schedule over budget; coarse shared dispatch tick (≤15-min, not a per-task cron/long sleep); owner kill-switch/pause on any task. | ✅ (MAX_ACTIVE / per-run cost ceiling) |
| 11 | **Inngest 50k-exec Free cap exhausted → Hobby PAUSES *all* execution** — reminders, scheduled-task dispatch *and* memory writes stop silently, not just ingest. | High · Med | Batch the ambient lane (5/30s) → ~5× fewer executions; daily-arm reminders + a coarse (≤15-min) shared scheduled-task dispatch cron (not per-minute); per-function counter + **external heartbeat** alert on stall; Neon is source-of-truth so nothing is lost, only delayed; `batchEvents` valve if volume climbs. | ✅ (reminder/dispatch cadence) |
| 12 | **Reminder / scheduled task never / late / double fires** — a single `sleepUntil` exceeds the **7-day Free sleep cap** (reminders *and* long-cadence monthly/interval tasks); or per-minute polling exhausts executions; or a cancellation race double-fires. | High · Med | Durable Neon `reminders` / `scheduled_tasks` tables as source-of-truth; **daily-arm + <6-day `sleepUntil`** re-arm for reminders; scheduled tasks fan out from a **horizon-unbounded dispatch scan over `next_run_at`, never a per-task long sleep**; independent catch-up cron sweeper; **atomic claim-before-send** (`UPDATE…WHERE status='armed' RETURNING` / `FOR UPDATE SKIP LOCKED`); `onFailure` reaper for stranded `firing`/`running` rows; test a 30-day-out reminder and a monthly task both fire via the scan. | ✅ (accept daily-arm/scan) |

| 14 | **Webhook data-loss vs retry-storm dilemma** — inline AI work → slow response → Telegram retry storm + duplicate processing; but acking 200 on an enqueue failure → permanently lost message. | High · Med | Handler = verify → dedupe → enqueue → return fast (sub-100ms, no LLM/heavy DB inline). Return **5xx on enqueue failure** so Telegram redelivers; `update_id` PK + `ON CONFLICT DO NOTHING` makes redelivery idempotent. (Verified tolerance ~60s, not 10s — fast-ack rule is conservatively correct.) | — |
| 15 | **Over-notification fatigue** — proactive pings / scheduled-task results / digests fire too often → the house mutes Baumy entirely (kills the product). | High · Med | Coalesce sub-threshold items into scheduled digests (primary lever); conservative caps (≤5/day to the group), quiet hours, per-reason cooldowns, silent-by-default, snooze/mute; **scheduled-task results and digests route through the same notify-gate** (user-requested results a bounded priority tier; task failures go to the admin, not the house); P0 reminders exempt so they're never swallowed. v1 reminders/nudges are **group-delivered, not per-user DM** (A3), so there is no DM-notification surface to over-fire. | ✅ (caps/cadence) |

## Tier 3 — Correctness & data integrity

| # | Risk | Sev · Likelihood | Mitigation | Owner? |
|---|------|------------------|-----------|:---:|
| 16 | **Timezone / DST mis-parse** fires reminders *or scheduled tasks* at the wrong absolute hour (chrono has no IANA; Vercel runs UTC; Berlin is GMT+1 winter / GMT+2 summer). | High · Med | Hybrid parse: chrono/cadence-phrase → Luxon/croner IANA wall-clock→UTC using the **target** occurrence's offset; store `timestamptz`; dispatch crons pinned `TZ=Europe/Berlin`, avoid the 01:00–03:00 DST window / add jitter; **echo the resolved local time in the confirm** so a housemate catches errors; golden tests spanning a Berlin DST transition. | ✅ (house tz = `Europe/Berlin`) |
| 17 | **Hard-coded `temperature` on a pinned Opus ≥4.7 id → 100% HTTP-400 outage** on the quality/advisor path (now also the deliberative web-search runs). | High · Low | Never set `temperature`/`top_p`/`top_k` for Anthropic quality models; gate sampling behind `modelAcceptsSampling` (unknown → omit); store `send_temperature=false` for Opus; unit-test the guard; the deliberative + scheduled-task advisor runs use the same guard. | — |
| 18 | **Duplicate reply / reminder / scheduled-task run** posted after a crash between send-success and step-completion (no Telegram idempotency key). | Med · Med | Claim-before-send (`INSERT…update_id ON CONFLICT DO NOTHING RETURNING`; `FOR UPDATE SKIP LOCKED` + `(task_id, slot)` UNIQUE for tasks); reply/send step `retries:0`; accept at-most-once for the group; all fire paths route through one atomic claim. | — |
| 19 | **Non-deterministic extraction not isolated in its own step** — a retry re-runs the LLM and yields different facts, corrupting `(update_id, ordinal)` memory. | Med · Med | Mandate `classify-extract` as a dedicated memoized `step.run`; the write step consumes only memoized output; test the LLM is called once across a simulated retry. | — |
| 20 | **Right-to-be-forgotten not honored** — soft-delete leaves plaintext, vector, *and secure-value ciphertext* in-table and in backups. | Med · Low | Overwrite `content` + null `embedding` + **drop `content_encrypted`** on tombstone; Inngest hard-purge after grace; deliberate `onDelete` per FK (CASCADE owned, SET NULL provenance); document backup-rotation and that **destroying/rotating the encryption key** is itself a full-erasure lever for `secure_value` facts. | ✅ (retention) |
| 21 | **Single-tenant assumptions baked in → cross-group leak or a future rewrite** — v1 hosts only this one house's data (single-tenant; hosting others is an explicit non-goal). If `group_id` origin-scoping is skipped as "unnecessary for one group", a later 2nd group (the owner running another house) silently leaks house-A memory into house-B answers — or the opposite over-reaction, building multi-tenant SaaS the owner explicitly does not want. | Med · Low | Add the `group_id`/origin-scope column **now** as cheap hygiene (D-sec) and scope every retrieval + the audited grounding function by it, so multi-group is an additive flip, not a rewrite; but **do NOT build multi-tenant SaaS** (non-goal Q0) — no per-tenant isolation/RLS/billing in v1; trust boundary stays the single `BAUMY_HOUSE_CHAT_ID`. | ✅ (confirm v1 single-group) |
| 22 | **App-side encryption key mishandling** (new capability) — the `secure_value` subset (door/gate/alarm codes, wifi, bank details) is AES-256-GCM sealed under `BAUMY_ENCRYPTION_KEY`; the key landing in the DB/logs decrypts every secret, a lost/rotated key permanently orphans those facts, or a secret gets embedded / decrypted into a digest. | Med · Low | Key lives **only** in `BAUMY_ENCRYPTION_KEY` (Vercel secret) — **never** in the DB/backups/logs, so a DB dump alone is inert; assert exactly 32 bytes at boot; **embed the non-secret descriptor, never the plaintext** (`embedding` computed from the descriptor only); `openSecureValue` is called **only** by the answer executor for a direct member ask — **never** for digests/broadcasts/nudges/deliberative context; never log or serialize plaintext; document a rotation/re-encrypt story and that "lost key" ⇒ those facts are permanently unreadable (cross-ref RTBF, risk 20). | ✅ (key provisioning + rotation) |
| 23 | **Entity-resolution drift** — one thing ("the printer"/"Prusa"/"MK4") fragments across rows, silently breaking recall. | Med · Med | Match-then-merge using name-embedding cosine + alias/trgm overlap with a conservative threshold; prefer a new entity over a false-merge; periodic merge sweep. | — |
| 24 | **Out-of-order processing** corrupts sequencing — a *verification refuted* the claim that `max_connections=1` guarantees global-serial delivery; Telegram only guarantees ordering by increasing `update_id` and admits out-of-order. | Low · Med | If strict per-chat ordering matters, enforce **app-side** (process by `update_id` / per-chat `concurrency:{key:chat_id,limit:1}`), not via `max_connections`; otherwise design writes order-independent (timestamped provenance upsert). | — |

---

### The through-line
Tiers 1–3 are not independent bugs — they are the **same architectural spine** seen from different angles: *untrusted input never authorizes a privileged effect; every effect is deterministic, templated, and destination-fixed; the LLM only ever proposes.* The three new lanes bolt onto that spine rather than bypass it — the **web-search lane** is explicit-trigger-only, secret-starved, and sink-pinned; **scheduled tasks** are write-gated to create and hard-bounded to run; **self-config** is gated, direction-capped, and dashboard-reversible. Build the spine once (deterministic write-gate + code-derived trust tiers + fixed `sendToHouse` + `group_id` origin scope + spend ledger + app-held encryption key) and the majority of the register is mitigated structurally rather than case-by-case.
